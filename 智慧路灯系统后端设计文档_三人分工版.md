# 智慧路灯节能系统后端设计文档（三人分工版）

**项目名称：** 智慧路灯节能系统后端  
**文档版本：** V1.0  
**适用范围：** 后端开发、接口联调、课程答辩、项目展示  
**主要技术栈：** Python、FastAPI、SQLAlchemy、MySQL、MQTT、APScheduler、JWT、Pydantic  

---

## 一、项目概述

智慧路灯节能系统后端用于管理智慧路灯设备、接收设备上报的光照数据、根据阈值生成控制建议、支持手动控制路灯、记录控制日志、监测设备在线状态、生成离线告警，并提供运维智能问答能力。

系统后端通过 HTTP 接口向前端或测试工具提供设备管理、数据查询、阈值配置、控制命令、告警查询和智能体问答能力；通过 MQTT 与模拟设备或真实设备进行数据通信，实现 telemetry 数据上报、status 心跳上报和 command 控制命令下发。

系统整体目标是形成一个完整的“设备管理—数据采集—自动联动—手动控制—状态监测—离线告警—智能问答”的后端闭环。

## 二、系统功能需求

### 2.1 设备管理功能

设备管理模块用于维护智慧路灯设备的基础信息。系统需要支持设备新增、查询、修改和删除功能，并保证设备编码唯一。

主要功能包括：

1. 新增路灯设备。
2. 查询设备列表。
3. 查询单个设备详情。
4. 修改设备名称、安装位置、设备状态等信息。
5. 删除设备。
6. 维护设备在线状态和最后心跳时间。

### 2.2 光照数据采集功能

系统需要接收设备上报的光照数据，并支持实时查询和历史查询。

主要功能包括：

1. 通过 HTTP 接口模拟设备上报光照数据。
2. 通过 MQTT telemetry 主题接收设备上报数据。
3. 保存光照强度、路灯状态、电压、上报时间等信息。
4. 查询某个设备的最新光照数据。
5. 查询某个设备的历史光照数据。

### 2.3 阈值配置功能

系统需要为每个设备配置光照判断阈值，用于自动判断是否需要开灯或关灯。

主要功能包括：

1. 查询设备阈值配置。
2. 若设备没有阈值配置，自动创建默认配置。
3. 修改低阈值、高阈值和是否启用自动判断。
4. 校验低阈值必须小于高阈值。

### 2.4 手动控制功能

系统需要支持后台向路灯下发控制命令。

主要功能包括：

1. 手动开灯。
2. 手动关灯。
3. 设置亮度。
4. 记录控制日志。
5. 在 MQTT 开启时向 command 主题发布控制命令。
6. MQTT 未启用或发布失败时，接口不能崩溃，需要记录对应结果。

### 2.5 自动联动功能

系统需要根据光照数据和阈值配置生成自动判断建议。

基本判断规则：

1. 光照值低于低阈值时，系统建议开灯。
2. 光照值高于高阈值时，系统建议关灯。
3. 光照值处于阈值区间内时，系统建议保持当前状态。
4. 阈值配置关闭时，返回禁用或不执行自动判断结果。

### 2.6 MQTT 通信功能

系统需要通过 MQTT 与设备进行通信。

主要功能包括：

1. 订阅 telemetry 主题，接收光照数据并写入 `light_data` 表。
2. 订阅 status 主题，接收设备心跳并更新在线状态。
3. 发布 command 主题，向设备下发控制命令。
4. 支持在 `MQTT_ENABLED=false` 时关闭 MQTT，不影响 HTTP 接口测试。

### 2.7 设备心跳与在线状态功能

系统需要判断设备是否在线。

主要功能包括：

1. 设备通过 MQTT status 主题上报心跳。
2. 后端更新设备状态和最后心跳时间。
3. 定时任务检查设备最后心跳时间。
4. 超过离线阈值后，将设备状态改为 `offline`。

### 2.8 离线告警功能

系统需要在设备离线时生成告警日志。

主要功能包括：

1. 设备离线后自动生成离线告警。
2. 查询告警列表。
3. 查询告警详情。
4. 标记告警为已处理。
5. 避免同一设备重复生成大量未处理离线告警。

### 2.9 智能体问答功能

系统需要提供运维问答接口，根据系统已有数据生成维护建议。

主要功能包括：

1. 根据设备 ID 或设备编码查询设备上下文。
2. 不指定设备时生成系统整体运维分析。
3. 汇总设备状态、最新光照、阈值配置、控制日志和告警日志。
4. 在未启用大模型时返回规则版回答。
5. 在大模型配置缺失或调用失败时返回 fallback 回答，保证接口不崩溃。
6. 智能体只提供建议，不直接执行开灯、关灯、修改阈值、处理告警或删除数据。

## 三、系统总体架构设计

### 3.1 架构说明

系统采用分层后端架构，主要包括接口层、数据校验层、业务服务层、数据模型层、数据库层、MQTT 通信层和定时任务层。

```text
前端 / Swagger / Postman / 测试脚本
        ↓ HTTP JSON
FastAPI Router 接口层
        ↓
Schema 参数校验层
        ↓
Service 业务逻辑层
        ↓
Model 数据模型层
        ↓
MySQL 数据库

模拟设备 / MQTTX / 真实设备
        ↓ MQTT
MQTT Client + Handlers
        ↓
Service 业务逻辑层
        ↓
MySQL 数据库

APScheduler 定时任务
        ↓
心跳检测 / 离线告警
        ↓
MySQL 数据库
```

### 4.1 毕超荃：设备管理 + 阈值配置 + 手动控制模块负责人

#### 4.1.1 职责定位

毕超荃负责系统的基础设备管理和人工控制能力，保证平台能够管理路灯设备，并通过接口控制路灯开关。

#### 4.1.2 负责功能

1. 设备管理 CRUD。
2. 阈值配置。
3. 手动开灯、关灯、设置亮度。
4. 控制日志基础记录。
5. 路由注册与接口联调。

#### 4.1.3 主要代码文件

```text
backend/app/routers/devices.py
backend/app/routers/thresholds.py
backend/app/routers/commands.py
backend/app/models/device.py
backend/app/models/threshold_config.py
backend/app/schemas/device.py
backend/app/schemas/threshold_config.py
backend/app/services/control_service.py
```

#### 4.1.4 模块实现思路

设备管理模块通过 `devices.py` 提供新增、查询、修改和删除接口。新增设备时，系统校验 `device_code` 是否重复；查询设备时返回设备编号、设备名称、位置、状态和最后心跳时间；修改设备时支持部分字段更新；删除设备时根据设备 ID 删除对应记录。

阈值配置模块通过 `thresholds.py` 提供查询和修改接口。每个设备对应一条阈值配置，包含低阈值、高阈值和是否启用自动判断。当设备没有阈值配置时，系统自动创建默认配置。修改阈值时需要校验 `low_threshold < high_threshold`。

手动控制模块通过 `commands.py` 接收控制命令，支持 `TURN_ON`、`TURN_OFF` 和 `SET_BRIGHTNESS`。接口接收到命令后调用 `control_service.py`，由服务层负责封装 MQTT command payload、发布控制命令并写入 `control_logs` 表。若 MQTT 未启用，控制日志记录为 `skipped`；若发布失败，控制日志记录为 `failed`，接口本身不应直接崩溃。

#### 4.1.5 答辩说明

答辩时可以说明：

```text
我主要负责设备管理、阈值配置和手动控制模块。设备管理模块负责维护系统中的路灯设备基础信息，阈值配置模块用于设置每个设备的自动判断标准，手动控制模块用于管理员直接向设备下发开灯、关灯或亮度调节命令，并将每次控制操作写入控制日志，方便后续追踪和排查。
```

#### 4.2.4 模块实现思路

光照数据模块通过 `light_data.py` 提供 HTTP 上报、最新光照查询和历史光照查询接口。HTTP 上报接口用于模拟设备直接向后端提交光照数据，提交成功后写入 `light_data` 表，并调用自动联动逻辑生成 `suggested_action`。

自动联动模块通过 `auto_control.py` 实现判断逻辑。系统根据设备当前光照值和阈值配置进行判断：低于低阈值建议开灯，高于高阈值建议关灯，处于阈值范围内则保持当前状态。该模块为后续真实自动控制或控制命令下发提供基础。

MQTT 接入模块通过 `mqtt/client.py` 连接 MQTT Broker，通过 `mqtt/handlers.py` 处理设备发布的 telemetry 消息。设备发布到 `streetlight/{deviceId}/telemetry` 后，后端解析 JSON payload，校验 `deviceId` 是否存在，存在则将数据写入 `light_data` 表，并执行自动判断逻辑。

模拟设备程序位于 `backend/mock_device/`，用于在没有真实硬件时定时生成光照数据，通过 MQTT 或 HTTP 方式上报后端，支撑项目演示和联调测试。

### 4.3 唐岩：设备心跳 + 在线状态 + 离线告警 + 智能问答模块负责人

3. 离线告警生成。
4. 告警日志查询。
5. 智能问答接口。
6. 测试脚本和最终演示流程。


#### 4.3.4 模块实现思路

设备心跳模块通过 MQTT status 主题接收设备在线状态。设备向 `streetlight/{deviceId}/status` 发布消息后，后端根据 `online` 字段更新设备状态，并将 `timestamp` 写入 `devices.last_heartbeat_at`。如果设备没有携带时间戳，则以后端当前时间作为最后心跳时间。

心跳检测任务通过 `heartbeat_check.py` 定时执行。系统只检查当前状态为 `online` 且存在 `last_heartbeat_at` 的设备。如果当前时间与最后心跳时间的间隔超过 `DEVICE_OFFLINE_SECONDS`，系统将设备状态改为 `offline`，并调用告警服务生成离线告警。

离线告警模块通过 `alarm_service.py` 和 `alarms.py` 实现。告警服务负责在设备离线后创建 `offline` 告警，并避免同一设备重复生成未处理离线告警。告警接口支持查询告警列表、查询告警详情和标记告警已处理。

智能问答模块通过 `agent.py` 和 `agent_service.py` 实现。接口接收用户问题后，根据 `device_id` 或 `device_code` 查询设备上下文；若不指定设备，则查询系统整体上下文。上下文包括设备状态、最新光照、历史光照、阈值配置、控制日志和告警日志。系统根据这些数据生成运维建议。在 `LLM_ENABLED=false` 时使用规则版回答；在大模型启用但调用失败时返回 fallback 回答，保证接口可用。

#### 4.3.5 答辩说明

答辩时可以说明：

```text
我主要负责设备心跳、在线状态、离线告警和智能问答模块。设备通过 MQTT status 主题上报心跳，后端记录最后心跳时间。定时任务会检查设备是否超时未上报，如果超时则将设备标记为离线，并生成离线告警。智能问答接口会根据设备状态、光照数据、控制日志和告警日志生成运维建议，帮助管理员快速判断设备问题。
```

## 五、数据库设计

### 5.1 数据库总体说明

系统数据库使用 MySQL，数据库名为 `smart_streetlight`。主要保存用户、设备、光照数据、阈值配置、控制日志和告警日志。

主要数据表包括：

| 表名 | 作用 |
|---|---|
| `users` | 保存登录用户、密码哈希和角色信息 |
| `devices` | 保存路灯设备基础信息 |
| `light_data` | 保存设备光照和状态上报数据 |
| `threshold_configs` | 保存设备自动判断阈值配置 |
| `control_logs` | 保存手动控制指令日志 |
| `alarm_logs` | 保存设备离线等告警日志 |

### 5.2 devices 表

`devices` 表用于保存路灯设备基础信息。

| 字段 | 类型 | 说明 |
|---|---|---|
| id | int | 设备主键 ID |
| device_code | varchar(64) | 设备编码，唯一，例如 `SL-001` |
| device_name | varchar(100) | 设备名称，例如 `一号路灯` |
| location | varchar(255) | 安装位置 |
| status | varchar(20) | 设备状态，例如 `online`、`offline` |
| last_heartbeat_at | datetime | 最后一次心跳时间 |
| created_at | datetime | 创建时间 |
| updated_at | datetime | 更新时间 |

### 5.3 light_data 表

`light_data` 表用于保存设备光照和状态上报数据。

| 字段 | 类型 | 说明 |
|---|---|---|
| id | int | 光照数据 ID |
| device_id | int | 关联设备 ID |
| light_intensity | int | 光照强度 |
| lamp_status | varchar(20) | 路灯状态，例如 `on`、`off` |
| voltage | float | 电压 |
| reported_at | datetime | 设备上报时间 |
| created_at | datetime | 后端入库时间 |

### 5.4 threshold_configs 表

`threshold_configs` 表用于保存每台设备的自动判断阈值。

| 字段 | 类型 | 说明 |
|---|---|---|
| id | int | 阈值配置 ID |
| device_id | int | 关联设备 ID，一个设备一条配置 |
| low_threshold | int | 低阈值，低于该值建议开灯 |
| high_threshold | int | 高阈值，高于该值建议关灯 |
| enabled | boolean | 是否启用自动判断 |
| updated_at | datetime | 更新时间 |

### 5.5 control_logs 表

`control_logs` 表用于保存控制命令执行记录。

| 字段 | 类型 | 说明 |
|---|---|---|
| id | int | 控制日志 ID |
| device_id | int | 关联设备 ID |
| command | varchar(50) | 控制命令，例如 `TURN_ON`、`TURN_OFF`、`SET_BRIGHTNESS` |
| source | varchar(20) | 命令来源，例如 `manual` |
| result | varchar(20) | 执行结果，例如 `success`、`failed`、`skipped` |
| request_payload | json | 控制请求内容 |
| reply_payload | json | 设备回复内容，当前可为空 |

## 七、MQTT 通信设计

### 7.1 MQTT 配置

MQTT 配置通过 `backend/.env` 读取。

```env
MQTT_ENABLED=false
MQTT_HOST=127.0.0.1
MQTT_PORT=1883
MQTT_USERNAME=
MQTT_PASSWORD=
MQTT_CLIENT_ID=smart-streetlight-backend
```

设备离线检测配置：

```env
SCHEDULER_ENABLED=true
DEVICE_OFFLINE_SECONDS=180
```

### 7.2 Topic 命名规则

```text
streetlight/{deviceId}/{messageType}
```

其中 `deviceId` 使用设备编码，对应数据库 `devices.device_code`，例如 `SL-001`。

### 7.3 Topic 列表

| Topic | 方向 | 说明 |
|---|---|---|
| `streetlight/{deviceId}/telemetry` | 设备 -> 后端 | 上报光照、路灯状态、电压等遥测数据 |
| `streetlight/{deviceId}/status` | 设备 -> 后端 | 上报在线状态和心跳 |
| `streetlight/{deviceId}/command` | 后端 -> 设备 | 下发开灯、关灯、亮度调节等控制命令 |
| `streetlight/{deviceId}/command/reply` | 设备 -> 后端 | 后续用于返回控制结果 |
| `streetlight/{deviceId}/alarm` | 设备 -> 后端 | 后续用于设备主动上报告警 |

### 7.4 Telemetry 上报流程

```text
设备发布 telemetry 消息
↓
后端 MQTT handler 接收消息
↓
解析 JSON payload
↓
根据 deviceId 查询 devices 表
↓
设备存在则写入 light_data 表
↓
执行自动联动判断
```

### 7.5 Status 心跳流程

```text
设备发布 status 消息
↓
后端解析 online 和 timestamp
↓
online=true 时更新设备为 online
↓
online=false 时更新设备为 offline
↓
更新 devices.last_heartbeat_at
```

### 7.6 Command 下发流程

```text
用户调用手动控制接口
↓
后端校验设备和命令参数
↓
封装 command payload
↓
发布到 streetlight/{deviceCode}/command
↓
写入 control_logs 控制日志
```

## 八、权限设计

### 8.1 角色设计

系统设计三类用户角色：

| 角色 | 说明 | 权限范围 |
|---|---|---|
| admin | 管理员 | 访问全部接口，包括设备管理、控制命令、告警处理 |
| operator | 运维人员 | 查看数据、上报光照、修改阈值、下发控制命令、处理告警 |
| viewer | 只读用户 | 查看设备、光照、阈值、控制日志、告警和智能体问答 |

### 8.2 认证流程

```text
用户登录
↓
后端校验用户名和密码哈希
↓
校验成功后生成 JWT access token
↓
前端保存 token
↓
后续请求携带 Authorization: Bearer <access_token>
↓
后端解析 token 并判断角色权限
```

### 8.3 权限控制说明

公共接口不需要 Token：

```text
GET /
GET /api/health
POST /api/auth/init-admin
POST /api/auth/login
```

业务接口需要 Token。设备新增、修改、删除仅允许 `admin`；光照上报、阈值修改、控制命令和告警处理允许 `admin` 和 `operator`；查看类接口和智能问答允许三种角色访问。

## 九、智能体问答设计

### 9.1 智能体定位

智能体定位为智慧路灯运维问答 Agent，面向项目演示和运维排查。它根据后端已有数据回答设备状态、光照、阈值、控制日志和告警相关问题。

### 9.2 问答数据来源

设备维度上下文包括：

1. `devices`：设备基础信息、状态、最后心跳时间。
2. `light_data`：最新光照、最近光照历史。
3. `threshold_configs`：阈值配置。
4. `control_logs`：最近控制日志。
5. `alarm_logs`：最近告警日志、未处理告警数量。

系统维度上下文包括：

1. 设备总数。
2. 在线设备数。
3. 离线设备数。
4. 未处理告警数量。
5. 最近告警。
6. 最近控制日志。

### 9.3 问答处理流程

```text
用户提交 question
↓
判断是否指定 device_id 或 device_code
↓
查询设备维度或系统维度上下文
↓
根据上下文生成规则版回答或调用大模型
↓
返回 answer、source、related_device 和 context_summary
```

### 9.4 安全约束

智能体只能提供运维建议，不能直接执行以下操作：

1. 开灯、关灯或设置亮度。
2. 修改阈值配置。
3. 处理告警。
4. 删除设备或业务数据。
5. 修改数据库。
6. 替代用户执行控制命令。

## 十、测试与联调设计

### 10.1 测试前准备

### 10.2 推荐测试顺序

1. `GET /api/health` 健康检查。
2. `POST /api/devices` 创建设备。
3. `GET /api/devices` 查询设备列表。
4. `POST /api/devices/{device_id}/light-data` 上报光照。
5. `GET /api/devices/{device_id}/latest-light` 查询最新光照。
6. `GET /api/devices/{device_id}/light-history` 查询历史光照。
7. `GET /api/devices/{device_id}/threshold` 查询阈值。
8. `PUT /api/devices/{device_id}/threshold` 修改阈值。
9. 再次上报光照，观察 `suggested_action`。
10. `POST /api/devices/{device_id}/commands` 下发手动控制命令。
11. `GET /api/devices/{device_id}/commands` 查询控制日志。
12. MQTTX 发布 telemetry，验证光照入库。
13. MQTTX 发布 status，验证设备状态和心跳时间更新。
14. 等待离线检测任务触发，验证设备离线和告警生成。
15. `GET /api/alarms` 查询告警。
16. `PUT /api/alarms/{alarm_id}/handle` 标记告警已处理。
17. `POST /api/agent/chat` 查询智能体运维建议。

### 10.3 测试重点

| 模块 | 测试重点 |
|---|---|
| 设备管理 | CRUD 是否正常，重复 `device_code` 是否返回 400 |
| 光照数据 | 数据是否能写入，最新和历史查询是否正确 |
| 阈值配置 | 默认配置是否自动创建，阈值范围校验是否生效 |
| 控制命令 | 命令是否合法，控制日志是否写入，MQTT 发布失败是否不崩溃 |
| MQTT | telemetry 是否入库，status 是否更新设备状态，command 是否能被订阅端收到 |
| 心跳检测 | 超时设备是否变为 offline |
| 告警管理 | 离线后是否生成告警，是否能标记已处理 |
| 智能体问答 | 是否能根据设备或系统上下文生成建议，异常情况下是否 fallback |
