# 真实大模型接入说明

## 1. 当前后端是否已经支持真实大模型

支持。

当前后端已经实现了智能问答接口的真实大模型接入能力，调用入口为：

```text
POST /api/agent/chat
```

当前实现方式为：

- 使用 `OpenAI-compatible` 接口风格
- 后端会先整理设备/系统上下文
- 再把用户问题和上下文一起发给大模型
- 如果调用成功，接口返回：

```json
{
  "source": "llm"
}
```

如果没有配置或调用失败，则会退回：

- `source=mock`
- 或 `source=fallback`

相关代码：

- `backend/app/core/config.py`
- `backend/app/services/llm_client.py`
- `backend/app/routers/agent.py`

---

## 2. 当前需要你补的只有哪几项

你现在只需要补本地 `backend/.env` 中这 5 项：

```env
LLM_ENABLED=true
LLM_PROVIDER=openai-compatible
LLM_API_KEY=你的真实密钥
LLM_BASE_URL=你的兼容接口基础地址
LLM_MODEL=你的模型名
LLM_TIMEOUT_SECONDS=30
```

说明：

- `LLM_ENABLED=true`：开启真实大模型
- `LLM_PROVIDER=openai-compatible`：当前后端只实现了这种兼容风格
- `LLM_API_KEY`：你的平台发给你的密钥
- `LLM_BASE_URL`：一定要填到 `/v1` 这一层，不要写 `/chat/completions`
- `LLM_MODEL`：你准备使用的模型名
- `LLM_TIMEOUT_SECONDS`：超时时间，默认 `30`

---

## 3. 关键注意点

### 3.1 `LLM_BASE_URL` 不能写错

后端代码会自动拼接：

```text
{LLM_BASE_URL}/chat/completions
```

所以你应该填：

```env
LLM_BASE_URL=https://api.openai.com/v1
```

而不是：

```env
LLM_BASE_URL=https://api.openai.com/v1/chat/completions
```

### 3.2 不要把真实 API Key 提交到 GitHub

`LLM_API_KEY` 只能放在你自己的本地：

- `backend/.env`

不要提交到：

- `.env.example`
- `README`
- Git 仓库

---

## 4. 一份可直接填写的模板

你可以直接把 `backend/.env` 中的 LLM 配置改成下面这样：

```env
LLM_ENABLED=true
LLM_PROVIDER=openai-compatible
LLM_API_KEY=你的真实APIKey
LLM_BASE_URL=你的OpenAI兼容平台地址/v1
LLM_MODEL=你的模型名
LLM_TIMEOUT_SECONDS=30
```

例如如果你使用标准 OpenAI 风格平台，形式通常类似：

```env
LLM_ENABLED=true
LLM_PROVIDER=openai-compatible
LLM_API_KEY=sk-xxxxxxxx
LLM_BASE_URL=https://api.openai.com/v1
LLM_MODEL=gpt-4o-mini
LLM_TIMEOUT_SECONDS=30
```

如果你用的是其他兼容平台，也同理：

- `LLM_BASE_URL` 填该平台提供的 OpenAI-compatible `/v1`
- `LLM_MODEL` 填平台支持的模型名称

## 6. 如何验证是否真的接上了真实大模型

### 6.1 用 Swagger

打开：

```text
http://127.0.0.1:8000/docs
```

调用：

```text
POST /api/agent/chat
```

请求示例：

```json
{
  "question": "SL-001 离线后应该怎么排查？",
  "device_code": "SL-001"
}
```

重点看返回里的：

```json
"source": "llm"
```

如果是：

- `llm`：说明已经走真实大模型
- `fallback`：说明启用了 LLM，但配置有问题或调用失败
- `mock`：说明当前还没启用真实大模型

### 6.2 用 PowerShell 测试

先登录：

```powershell
$body = @{ username = "admin"; password = "123456" } | ConvertTo-Json
$login = Invoke-RestMethod "http://127.0.0.1:8000/api/auth/login" -Method Post -ContentType "application/json" -Body $body
$headers = @{ Authorization = "Bearer $($login.access_token)" }
```

再提问：

```powershell
$payload = @{
  question = "SL-001 离线后应该怎么排查？"
  device_code = "SL-001"
} | ConvertTo-Json

Invoke-RestMethod "http://127.0.0.1:8000/api/agent/chat" -Method Post -Headers $headers -ContentType "application/json" -Body $payload
```

## 7. 常见失败原因

### 7.1 返回 `source=fallback`

常见原因：

1. `LLM_API_KEY` 没填
2. `LLM_BASE_URL` 没填
3. `LLM_MODEL` 没填
4. `LLM_BASE_URL` 写成了完整 `/chat/completions`
5. 平台网络不可达
6. 模型名写错
7. API Key 无权限

### 7.2 返回内容还是很像规则版

先看 `source`：

- 如果是 `mock`，说明根本没走真实大模型
- 如果是 `fallback`，说明真实大模型调用失败，后端退回规则版
- 只有 `source=llm` 才表示真正使用了外部大模型

### 7.3 后端没有报错但回答没变化

优先检查：

1. 修改 `.env` 后是否重启了后端
2. 当前访问的是否是最新启动的后端进程
3. 前端请求是否真的走到了 `/api/agent/chat`
4. 接口响应里的 `source` 是什么

## 8. 推荐接入顺序

建议按下面顺序操作：

1. 先准备一个可用的 OpenAI-compatible 平台
2. 拿到：
   - API Key
   - Base URL
   - Model 名称
3. 填入 `backend/.env`
4. 重启后端
5. 用 Swagger 测 `POST /api/agent/chat`
6. 确认返回 `source=llm`
7. 再去前端智能问答页面验证
