## 1. 项目背景

本项目来源于重庆交通大学与中软国际企业实训安排，实训主题为“端-智-云一体化软件综合设计”。项目选题为“智慧路灯节能系统”，目标是在 25 天实训周期内完成一个可运行、可演示、可答辩的智慧路灯应用。

项目分为两个阶段：

1. 校内 15 天软件阶段：不依赖真实硬件，先使用模拟设备产生光照、设备状态、心跳等数据，通过 MQTT 上报后端，完成数据接入、存储、告警、控制、前端可视化和智能体问答。
2. 基地 10 天硬件阶段：接入鸿蒙开发板、光照传感器和路灯控制外设，将模拟数据链路切换为真实硬件链路，完成端-智-云一体化联调。

项目第一版遵循“先软件闭环、后硬件落地”的原则，先打通“模拟数据 -> MQTT -> 后端 -> 数据库/告警/控制 -> 前端展示 -> 智能体问答”的主流程，再接入真实鸿蒙端设备。

## 2. 项目目标

### 2.1 总体目标

建设一套面向市政路灯管理场景的智慧路灯节能系统，实现路灯设备数据采集、光照监测、自动开关灯、远程控制、设备告警、可视化展示和智能维护问答。

### 2.2 具体目标

1. 实现光照强度、路灯开关状态、设备在线状态的实时监测。
2. 支持历史光照数据查询和趋势可视化。
3. 支持根据光照阈值自动控制路灯开关。
4. 支持管理员通过页面手动远程开灯、关灯。
5. 支持设备心跳监测和离线告警。
6. 支持路灯设备的添加、查看、绑定和解绑。
7. 支持告警记录查询和基础处置状态维护。
8. 接入智能体问答，提供路灯维护、故障排查、告警处理建议。
9. 校内阶段支持模拟数据运行，基地阶段支持真实鸿蒙端设备接入。
10. 形成完整的需求、设计、接口、测试、答辩和总结材料。

### 4.1 第一版必须完成

1. 光照强度实时监测。
2. 历史光照趋势展示。
3. 路灯自动联动控制。
4. 路灯手动开关控制。
5. 光照阈值设置。
6. 设备在线/离线状态监控。
7. 设备离线告警。
8. 路灯设备管理。
9. 告警日志查看。
10. 维护智能问答。

### 4.2 可选增强功能

1. 路灯亮度分级调节。
2. 路灯地图展示。
3. 告警处置流程和处置备注。
4. 智能体生成巡检建议。
5. 设备健康度评分。

4. 大规模设备压测和生产级高可用部署。
5. 与第三方市政平台的数据交换。

## 5. 业务流程

### 5.1 光照联动自动开关灯流程

1. 管理员在平台设置路灯自动控制阈值。
2. 设备端或模拟设备定时采集光照强度。
3. 设备通过 MQTT 上报光照数据和当前路灯状态。
4. 后端接收数据并保存到数据库。
5. 后端判断光照值是否低于开灯阈值或高于关灯阈值。
6. 若满足自动控制条件，后端生成控制指令并通过 MQTT 下发。
7. 设备执行开灯或关灯动作。
8. 设备回传执行结果，前端刷新路灯状态。
9. 系统记录控制日志，便于后续追溯。

### 5.2 手动远程控制流程

1. 管理员在设备详情页查看目标路灯状态。
2. 管理员点击“开灯”或“关灯”按钮。
3. 前端调用后端控制接口。
4. 后端校验设备状态并生成控制指令。
5. 后端通过 MQTT 向设备下发指令。
6. 设备执行指令并返回执行结果。
7. 前端展示执行成功或失败提示。
8. 后端记录手动控制日志。

### 5.3 设备离线告警流程

1. 设备定期发送心跳数据。
2. 后端记录设备最后一次心跳时间。
3. 当心跳中断超过设定时间后，系统判定设备离线。
4. 后端生成设备离线告警。
5. 前端告警列表和大屏告警统计同步更新。
6. 管理员查看告警并安排现场检查。
7. 设备恢复心跳后，系统更新设备在线状态。

### 5.4 维护智能问答流程

1. 管理员在智能问答页面输入问题。
2. 系统将问题发送给智能体服务。
3. 智能体从路灯维护知识库中检索相关资料。
4. 智能体生成维护建议或故障排查步骤。
5. 前端展示回答内容。
6. 项目演示时可结合离线告警、开关灯异常等场景进行问答。

## 9. 接口与通信需求

### 9.1 MQTT Topic

| Topic | 方向 | 用途 |
| --- | --- | --- |
| `streetlight/{deviceId}/telemetry` | 设备到云端 | 上报光照、路灯状态、心跳等数据 |
| `streetlight/{deviceId}/status` | 设备到云端 | 上报设备在线状态和路灯状态 |
| `streetlight/{deviceId}/command` | 云端到设备 | 下发开灯、关灯等控制指令 |
| `streetlight/{deviceId}/command/reply` | 设备到云端 | 回传控制指令执行结果 |
| `streetlight/{deviceId}/alarm` | 设备到云端或系统内部 | 上报告警或同步告警事件 |

## 10. 非功能需求

| 类别 | 需求 |
| --- | --- |
| 易用性 | 页面结构清晰，适合答辩演示；核心功能入口明显，操作流程不超过 3 步 |
| 实时性 | 模拟设备上报后，前端应在合理时间内刷新；校内演示建议控制在 3 秒以内 |
| 可靠性 | MQTT 断开、设备离线、控制超时等异常应有状态提示或日志记录 |
| 可维护性 | 前端、后端、模拟设备、智能体、硬件代码应分目录管理，接口字段统一维护 |
| 可扩展性 | 校内模拟数据源和基地真实硬件数据源应保持相同或相近的数据格式 |
| 安全性 | 控制接口应避免被任意调用；智能体回答不得编造危险操作；真实硬件阶段应注意用电和设备安全 |
| 可演示性 | 系统应支持通过一组固定演示数据快速展示实时监测、自动控制、手动控制、离线告警和智能问答 |

## 11. 技术约束

1. 数据传输优先采用 MQTT，前端实时刷新可采用 WebSocket、SSE 或短轮询。
2. 前端图表优先采用 ECharts。
3. 后端可采用 Spring Boot 或 Node.js，数据库可采用 MySQL 或 PostgreSQL。
4. 模拟设备可采用 Python 或 Node.js 脚本实现。
5. 智能体可采用 MaxKB 或企业/学校指定的大模型平台。
6. 基地阶段硬件以鸿蒙开发板、光照传感器、LED 或继电器模拟路灯为主要对象。
7. 校内阶段不得阻塞在真实硬件上，应优先保证模拟态完整闭环。

## 12. 验收标准

### 12.1 校内中期验收

1. 后端服务可以启动。
2. 前端页面可以启动。
3. Mock 设备可以持续上报光照和心跳数据。
4. 后端可以通过 MQTT 接收数据并保存。
5. 前端可以展示实时光照、历史光照曲线和设备状态。
6. 光照低于阈值时，系统可以自动产生开灯指令。
7. 页面可以手动开灯和关灯。
8. 设备心跳停止后，系统可以产生离线告警。
9. 智能体可以回答基础路灯维护问题。
10. 需求、设计、接口、测试文档基本完整。

### 12.2 基地终验

1. 鸿蒙端可以采集真实光照数据。
2. 真实设备数据可以上报到后端。
3. 前端可以展示真实设备数据。
4. 云端控制指令可以被鸿蒙端接收并执行。
5. 自动开关灯逻辑可以在真实或半真实硬件环境中跑通。
6. 系统具备真实态演示脚本、硬件接线图、设备清单和调试记录。
7. 最终答辩 PPT、演示视频或截图、源代码、测试记录和个人总结完整。

## 14. 风险与应对

| 风险 | 影响 | 应对措施 |
| --- | --- | --- |
| 真实硬件到位较晚 | 影响基地阶段联调 | 校内阶段先用 Mock 设备保证软件闭环，真实设备只替换数据源 |
| MQTT 链路不稳定 | 影响实时数据和控制 | 保留日志，先实现基础重连和失败提示，必要时用本地 broker 演示 |
| 前后端字段不一致 | 影响联调效率 | D2-D3 固定 API 字段和 MQTT 消息格式，变更需同步文档 |
| 智能体回答不稳定 | 影响演示效果 | 准备固定知识库和典型问题，答辩时使用稳定问题集 |
| 功能范围扩散 | 影响按时交付 | 第一版只做智慧路灯核心闭环，可选功能放到增强项 |
| 硬件控制存在安全风险 | 影响设备安全 | 使用低压 LED 或继电器模拟，接线前由硬件负责人确认 |

## 16. 第一版需求结论

第一版系统的核心不是追求大量功能，而是保证智慧路灯主流程能够完整跑通：

```text
模拟设备/真实硬件
-> MQTT 数据上报
-> 后端接收与存储
-> 阈值判断与告警生成
-> 控制指令下发
-> 前端实时展示
-> 智能体辅助维护
```

只要上述闭环稳定可演示，项目即可满足校内中期验收的主要要求；基地阶段再重点完成鸿蒙端真实数据接入和端-云控制联调。
