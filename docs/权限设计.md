# 权限设计

## 角色定义

系统采用三类角色，角色值统一存储在 `users.role` 字段中：

| 角色值 | 中文名称 | 权限说明 |
| --- | --- | --- |
| `admin` | 管理员 | 拥有全部页面和全部接口操作权限，包括用户管理、模拟器管理、设备管理、告警处理、阈值配置和手动控制。 |
| `maintainer` | 维修人员 | 可查看业务数据，可处理告警、下发路灯控制命令、修改阈值；不可管理用户、不可进入模拟器管理、不可新增/修改/删除设备基础信息。 |
| `user` | 普通用户 | 只读查看总览、设备、光照、告警和智能问答；不可执行写操作。 |

## 认证流程

1. 用户调用 `POST /api/auth/login` 登录。
2. 后端校验用户名、密码和账号启用状态。
3. 登录成功后返回 JWT 和当前用户信息。
4. 前端保存 token，并在后续请求中携带：

```text
Authorization: Bearer <token>
```

5. 后端通过 `get_current_user` 解析 token，并基于数据库中的当前用户状态和角色鉴权。

## 后端权限依赖

| 依赖函数 | 允许角色 |
| --- | --- |
| `get_current_user` | 已登录且启用的用户 |
| `require_admin` | `admin` |
| `require_maintainer_or_admin` | `admin` / `maintainer` |
| `require_user_or_above` | `admin` / `maintainer` / `user` |

## 前端控制

- 路由通过 `meta.roles` 控制访问范围，无权限访问跳转到 `/forbidden`。
- 侧边栏菜单根据当前角色过滤。
- 普通用户在设备详情页不能看到手动控制和保存阈值按钮。
- 普通用户在告警页不能看到处理告警按钮。
- 用户管理页和模拟器控制台仅管理员可访问。

注意：前端隐藏菜单和按钮只改善体验，真实安全边界以后端接口鉴权为准。

## 用户管理规则

- 管理员可创建用户、修改用户名、重置密码、修改角色、启用或禁用账号。
- 系统不提供硬删除用户接口，避免误删审计对象。
- 禁止禁用或降级最后一个启用状态的管理员。

## 旧数据迁移 SQL

如果数据库中已有旧角色值，请在部署新版代码前执行：

```sql
UPDATE users SET role = 'maintainer' WHERE role = 'operator';
UPDATE users SET role = 'user' WHERE role = 'viewer';
```

当前项目暂不引入 Alembic。`Base.metadata.create_all(bind=engine)` 只会创建不存在的表，不会自动修改已有表字段。

## 设备在线状态与告警确认口径

- 设备列表中的在线/离线来自数据库 `devices.status`，由真实心跳或 MQTT status 上报驱动。
- 模拟器列表中的运行状态只表示模拟器是否正在发 MQTT 数据，不等于设备在线/离线。
- 模拟器列表中的模拟上报状态表示下一次 status 消息会携带的 `online` 值。
- 模拟器列表中的系统设备状态表示数据库当前认定的设备状态，和设备列表保持一致。
- 停止模拟器不会立刻把设备改为离线；设备超过离线检测时间没有心跳后，定时任务才会标记离线并生成离线告警。
- 启动模拟器并成功上报 `online=true` 后，后端会把设备恢复为在线，并自动确认该设备未确认的离线告警。
- 告警页的“确认告警”只表示人工确认记录，不直接改变设备在线状态。
