# 智慧路灯节能系统项目分工与实施计划

## 1. 项目定位

本项目选题为“智慧路灯节能系统”，目标是在企业实训期间完成一个可演示、可联调、可答辩的端-智-云一体化应用。

项目分为两个阶段：

- 校内 15 天：先完成软件主线，不依赖真实硬件。使用模拟设备产生光照、设备状态等数据，通过 MQTT 上报到后端，再由前端大屏展示，并接入智能体问答与告警处理能力。
- 基地 10 天：接入真实鸿蒙开发板、光照传感器和路灯控制外设，把校内完成的软件系统切换到真实设备数据源，完成端-智-云联调。

第一阶段的核心原则是：先把“模拟数据 -> MQTT -> 后端 -> 数据库/告警/控制 -> 前端可视化 -> 智能体问答”跑通，再考虑真实硬件。

## 2. 第一版功能范围

第一版只做智慧路灯，不扩展到智慧农业、烟感、物流等其他场景。

必须完成的基础功能：

1. 光照强度实时监测：展示当前光照值。
2. 历史光照趋势：用折线图展示一段时间内的光照变化。
3. 路灯自动联动：光照低于阈值自动开灯，高于阈值自动关灯。
4. 路灯手动控制：页面按钮远程开灯、关灯。
5. 光照阈值设置：管理员可调整自动开关灯阈值。
6. 设备在线状态：展示路灯设备在线、离线状态。
7. 设备离线告警：设备心跳超时后生成离线告警。
8. 路灯设备管理：添加、查看、绑定、解绑路灯设备。
9. 告警日志查看：查看历史告警记录。
10. 维护智能问答：通过智能体回答路灯维护、故障排查相关问题。

可选增强功能：

- 路灯亮度分级调节。
- 设备地图展示。
- 告警处置流程。
- 智能体自动生成巡检建议。
- 基地阶段接入真实鸿蒙端控制指令。

## 3. 建议项目目录

建议仓库命名为 `smart-streetlight`，目录结构如下：

```text
smart-streetlight/
├── frontend/          # 前端页面、大屏、图表、设备控制界面
├── backend/           # 后端接口、数据库、MQTT 接入、告警与控制逻辑
├── mock-device/       # 模拟光照传感器、设备心跳、路灯状态上报脚本
├── agent/             # 智能体配置、知识库资料、维护问答材料
├── hardware/          # 鸿蒙端代码、接线图、设备清单、基地阶段记录
└── docs/              # 需求、设计、接口、测试、答辩、日报周报
```

文档目录建议至少包含：

```text
docs/
├── 需求说明.md

├── 数据库设计.md
```

## 4. 团队岗位分工

### 4.1 PM / 项目组长 1 人

主要职责：

- 维护项目范围，防止功能扩散。
- 制定每日任务和里程碑。
- 组织早会、晚会、评审会。
- 维护任务看板和风险清单。
- 汇总需求文档、设计文档、测试记录和答辩材料。
- 负责中期答辩和终验答辩的整体讲解逻辑。

当天必须产出：

- `docs/需求说明.md`
- 项目任务看板
- 每个人的任务清单

### 4.2 前端开发 2 人

前端 A：管理端页面与设备控制

- 登录后首页布局。
- 路灯设备列表。
- 设备详情页面。
- 手动开灯、关灯按钮。
- 阈值设置表单。
- 告警日志列表。

前端 B：可视化大屏与图表

- 实时光照卡片。
- 历史光照折线图。
- 设备在线/离线状态统计。
- 当前路灯状态展示。
- 告警数量统计。
- WebSocket 或 SSE 实时刷新。

当天必须产出：

- 页面清单。
- 页面原型草图。
- 前端路由设计。
- 需要后端提供的数据字段列表。

### 4.3 后端开发 3 人

后端 A：设备与数据接入

- MQTT 客户端接入。
- 接收模拟设备上报数据。
- 设计设备数据表。
- 提供实时数据和历史数据查询接口。

后端 B：控制与告警逻辑

- 光照阈值配置。
- 自动开关灯判断。
- 手动控制指令下发。
- 设备心跳检测。
- 离线告警生成。

后端 C：智能体与知识库接入

- 整理维护知识库。
- 接入 MaxKB 或其他智能体平台。
- 提供问答入口。
- 支持根据设备状态生成维护建议。

当天必须产出：

- 数据库表设计。
- API 接口清单。
- MQTT topic 设计。
- 后端模块分工。

### 4.4 硬件 / 鸿蒙端 1 人

校内阶段主要准备，基地阶段主导接入。

校内阶段职责：

- 明确后续真实硬件需要的传感器和控制对象。
- 学习鸿蒙开发板环境。
- 设计鸿蒙端与云端通信格式。
- 和后端约定真实设备上报字段。

基地阶段职责：

- 搭建鸿蒙开发环境。
- 接入光照传感器。
- 控制路灯或模拟继电器。
- 通过 MQTT 或 CoAP 上报数据。
- 订阅云端控制指令并执行。
- 编写接线图、设备清单和硬件调试记录。

当天必须产出：

- 硬件清单。
- 传感器数据字段说明。
- 控制指令说明。
- 基地阶段接入风险清单。

## 7. 技术选型建议

前端：

- Vue 3 或 React。
- ECharts 做图表。
- WebSocket 或 SSE 做实时刷新。

后端：

- Spring Boot 或 Node.js。
- MySQL 或 PostgreSQL。
- Redis 可选，用于缓存设备状态。
- MQTT broker 可用 Mosquitto、EMQX 或其他可用平台。

模拟设备：

- Python 或 Node.js 脚本。
- 定时发布光照值、设备心跳、路灯状态。

智能体：

- MaxKB 或其他学校/企业指定平台。
- 知识库内容包括设备维护、故障排查、光照联动规则、告警处理流程。

硬件：

- 鸿蒙开发板。
- 光照传感器。
- LED 或继电器模拟路灯。
- MQTT 或 CoAP 与云端通信。

## 8. MQTT Topic 初稿

建议先用以下 topic，后续可根据后端实现调整：

```text
streetlight/{deviceId}/telemetry     # 设备上报光照、状态、心跳
streetlight/{deviceId}/status        # 设备在线状态、路灯开关状态
streetlight/{deviceId}/command       # 云端下发控制指令
streetlight/{deviceId}/command/reply # 设备执行结果回传
streetlight/{deviceId}/alarm         # 设备告警
```

模拟设备上报数据示例：

```json
{
  "deviceId": "SL-001",
  "lightIntensity": 128,
  "lampStatus": "OFF",
  "online": true,
  "timestamp": "2026-07-01 14:00:00"
}
```

控制指令示例：

```json
{
  "deviceId": "SL-001",
  "command": "TURN_ON",
  "source": "manual",
  "timestamp": "2026-07-01 14:01:00"
}
```
