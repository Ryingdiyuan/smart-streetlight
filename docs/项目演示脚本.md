# 项目演示脚本

## 1. 演示准备

### 启动 MySQL

确认 MySQL 已启动，并存在数据库：

```sql
CREATE DATABASE IF NOT EXISTS smart_streetlight
DEFAULT CHARSET utf8mb4
COLLATE utf8mb4_unicode_ci;
```

如果是旧版 `devices` 表，确认已补充字段：

```sql
ALTER TABLE devices ADD COLUMN last_heartbeat_at DATETIME NULL;
```

展示时说明：

> 后端数据持久化保存到 MySQL，设备、光照、阈值、控制日志和告警日志都可以长期查询。

### 启动 Mosquitto

确认 MQTT Broker：

```text
Host: 127.0.0.1
Port: 1883
```

展示时说明：

> MQTT Broker 负责转发设备和后端之间的消息，设备不直接和后端 HTTP 接口绑定。

### 启动后端

```powershell
cd D:\code\smart-streetlight\backend
.\.venv\Scripts\uvicorn.exe app.main:app --reload
```

打开：

```text
http://127.0.0.1:8000/docs
```

### 检查 .env

本地演示建议：

```env
MQTT_ENABLED=true
LLM_ENABLED=false
```

如果只演示 HTTP 接口，可以设置：

```env
MQTT_ENABLED=false
```

## 2. 演示顺序

### 1. 健康检查

操作位置：Swagger `/docs`

接口：

```text
GET /api/health
```

预期结果：

```json
{
  "status": "ok"
}
```

展示时说明：

> 先确认后端服务正常运行。

### 2. 创建设备

接口：

```text
POST /api/devices
```

请求示例：

```json
{
  "device_code": "SL-001",
  "device_name": "一号路灯",
  "location": "校门口",
  "status": "offline"
}
```

预期结果：返回设备 ID。

展示时说明：

> 设备编码会作为 MQTT topic 中的 deviceId 使用。

### 3. HTTP 上报光照

接口：

```text
POST /api/devices/{device_id}/light-data
```

请求示例：

```json
{
  "light_intensity": 120,
  "lamp_status": "off",
  "voltage": 220.5,
  "reported_at": "2026-07-03T10:00:00"
}
```

预期结果：返回光照数据和 `suggested_action`。

展示时说明：

> HTTP 上报用于模拟设备数据，也方便本地快速测试。

### 4. MQTTX 发布 telemetry

操作位置：MQTTX

Topic：

```text
streetlight/SL-001/telemetry
```

Payload：

```json
{
  "deviceId": "SL-001",
  "lightIntensity": 80,
  "lampStatus": "off",
  "voltage": 220.5,
  "timestamp": "2026-07-03 10:00:00"
}
```

预期结果：后端写入 `light_data`。

展示时说明：

> 这一步展示设备通过 MQTT 上报遥测数据，后端订阅 topic 后入库。

### 5. 查询最新光照和历史光照

接口：

```text
GET /api/devices/{device_id}/latest-light
GET /api/devices/{device_id}/light-history
```

预期结果：能看到 HTTP 和 MQTT 上报后的光照数据。

展示时说明：

> 前端的实时状态卡片和光照曲线可以使用这些接口。

### 6. 配置阈值

接口：

```text
PUT /api/devices/{device_id}/threshold
```

请求示例：

```json
{
  "low_threshold": 100,
  "high_threshold": 300,
  "enabled": true
}
```

预期结果：返回阈值配置。

展示时说明：

> 阈值配置用于判断光照低时建议开灯、光照高时建议关灯。

### 7. 查看 suggested_action

再次上报光照：

```json
{
  "light_intensity": 80,
  "lamp_status": "off",
  "voltage": 220.5
}
```

预期结果：`suggested_action` 为 `TURN_ON`。

展示时说明：

> 当前自动判断只给建议，不直接控制设备，避免误操作。

### 8. 手动下发控制命令

接口：

```text
POST /api/devices/{device_id}/commands
```

请求示例：

```json
{
  "command": "TURN_ON"
}
```

预期结果：写入 `control_logs`；如果 MQTT 启用，会发布 command。

展示时说明：

> 手动控制会留下日志，便于追踪是谁在什么时候下发了什么命令。

### 9. MQTTX 订阅 command 查看命令

操作位置：MQTTX

订阅 Topic：

```text
streetlight/SL-001/command
```

预期结果：调用手动控制接口后，MQTTX 能收到控制命令。

展示时说明：

> 后端通过 MQTT 把控制命令发给设备侧。

### 10. 发布 status 心跳

Topic：

```text
streetlight/SL-001/status
```

Payload：

```json
{
  "deviceId": "SL-001",
  "online": true,
  "lampStatus": "off",
  "timestamp": "2026-07-03 10:00:00"
}
```

预期结果：设备状态和 `last_heartbeat_at` 更新。

展示时说明：

> 后端通过 status 心跳判断设备是否在线。

### 11. 等待离线检测生成告警

为了演示，可以把 `.env` 中：

```env
DEVICE_OFFLINE_SECONDS=30
```

重启后端后等待超过 30 秒。

预期结果：在线设备超时未上报心跳后，被标记为 `offline`，并生成离线告警。

展示时说明：

> APScheduler 会定时检查心跳时间，离线后写入告警表。

### 12. 查询告警并处理

接口：

```text
GET /api/alarms
PUT /api/alarms/{alarm_id}/handle
```

预期结果：能看到离线告警，并可标记为已处理。

展示时说明：

> 告警处理形成了从发现问题到人工确认的闭环。

### 13. 调用智能体问答

接口：

```text
POST /api/agent/chat
```

请求示例：

```json
{
  "question": "SL-001 最近为什么离线？",
  "device_code": "SL-001"
}
```

预期结果：返回设备状态、心跳、告警和排查建议。

展示时说明：

> 智能体只基于后端数据库上下文回答，不直接控制设备，因此适合做运维辅助。

## 3. 演示亮点

1. MQTT 模拟设备数据接入。
2. MySQL 持久化保存业务数据。
3. 阈值判断返回节能控制建议。
4. 手动控制命令通过 MQTT 下发。
5. 设备心跳和离线告警闭环。
6. 智能体根据系统上下文给出运维建议。

## 4. 演示注意事项

1. 不要在演示中展示真实数据库密码或 API Key。
2. 如果 MQTT 不稳定，可以先演示 HTTP 链路。
3. 如果大模型未配置，使用 `LLM_ENABLED=false` 的规则版智能体回答。
4. 智能体回答中不能声称已经执行控制操作。
