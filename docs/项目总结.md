# 项目总结

## 项目背景

智慧路灯节能系统是小组实训项目，目标是围绕路灯设备管理、环境光照采集、节能控制、设备状态监测和告警处理，搭建一套可演示、可扩展的后端服务。

当前阶段以后端为主，先使用模拟 HTTP/MQTT 数据打通核心链路，后续可接入前端页面、真实硬件和智能体问答。

## 第一阶段目标

第一阶段目标是完成后端基础能力：

1. 建立 FastAPI 后端项目骨架。
2. 使用 MySQL 保存设备和业务数据。
3. 支持设备管理、光照数据、阈值判断、控制命令和告警管理。
4. 接入 MQTT telemetry 和 status topic。
5. 准备完整接口文档、联调文档和测试材料。

## 技术选型

- Python 3.10+：语法清晰，适合快速开发后端服务。
- FastAPI：自动生成 Swagger 文档，适合前后端分离联调。
- MySQL：保存设备、光照、阈值、控制日志和告警日志。
- SQLAlchemy：使用 ORM 管理数据库表和对象。
- Paho-MQTT：接入 MQTT Broker，处理设备遥测和心跳。
- APScheduler：定时执行设备离线检测。
- pydantic-settings / python-dotenv：管理本地环境配置。

## 系统模块

1. 设备管理：设备 CRUD、设备状态和心跳时间。
2. 光照数据：HTTP 上报、最新数据查询、历史数据查询。
3. MQTT 遥测：订阅 telemetry topic 并写入光照数据表。
4. 阈值配置：低阈值、高阈值、是否启用自动判断。
5. 自动判断：根据光照和阈值返回建议动作。
6. 手动控制：通过 HTTP 接口记录控制日志，并可发布 MQTT command。
7. 心跳检测：通过 status topic 更新在线状态和最后心跳时间。
8. 离线告警：定时检测超时设备，写入告警日志。
9. 智能体问答：根据后端已有上下文生成运维建议。
10. 文档和测试：接口文档、联调流程、测试清单和冒烟测试脚本。

## 当前已跑通链路

### HTTP 设备 CRUD

前端或 Swagger 可以通过 `/api/devices` 完成设备新增、查询、修改和删除。

### HTTP 光照上报

通过 `/api/devices/{device_id}/light-data` 模拟设备上报光照数据，并保存到 MySQL。

### MQTT telemetry 入库

设备向 `streetlight/{deviceCode}/telemetry` 发布消息后，后端解析并写入 `light_data`。

### 阈值判断

光照数据写入后，后端根据设备阈值配置返回建议动作：`TURN_ON`、`TURN_OFF`、`KEEP` 或 `DISABLED`。

### 手动控制 command 发布

调用 `/api/devices/{device_id}/commands` 后，后端记录 `control_logs`，并在 MQTT 启用时向 `streetlight/{deviceCode}/command` 发布控制命令。

### status 心跳

设备向 `streetlight/{deviceCode}/status` 发布心跳后，后端更新设备 `status` 和 `last_heartbeat_at`。

### 离线告警

APScheduler 定时检查在线设备。如果最后心跳时间超过阈值，后端将设备标记为离线，并写入 `alarm_logs`。

### 智能体问答 MVP

调用 `/api/agent/chat` 后，后端根据设备或系统整体上下文整理数据，并返回运维建议。`LLM_ENABLED=false` 时可本地规则版演示；`LLM_ENABLED=true` 时可调用 OpenAI-compatible 大模型接口。

## 风险和注意事项

1. 当前数据库迁移仍依赖手动 SQL，`create_all` 不会修改已有表字段。
2. MQTT 测试依赖本地 Broker 正常运行。
3. 当前自动判断只返回建议动作，不直接替代人工控制。
4. 当前没有登录鉴权，接口默认用于实训和本地演示环境。
5. 智能体只提供排查建议，不直接执行控制命令或修改数据库。
