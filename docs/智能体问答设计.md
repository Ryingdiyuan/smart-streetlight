# 智能体问答设计

## 智能体定位

Day 11 的智能体定位为“智慧路灯运维问答 Agent”。它面向项目演示和运维排查，负责根据后端已有数据回答设备状态、光照、阈值、控制日志和告警相关问题。

## Day 11 实现范围

当前只实现一个核心接口：

```text
POST /api/agent/chat
```

能力范围：

1. 接收用户问题。
2. 根据 `device_id` 或 `device_code` 查询设备上下文。
3. 不指定设备时查询系统整体上下文。
4. 汇总设备信息、最新光照、历史光照、阈值配置、控制日志、告警日志。
5. 生成运维建议。
6. 支持 `LLM_ENABLED=false` 的本地规则版回答。
7. 支持 OpenAI-compatible chat completions 风格的大模型调用。

## Day 12 优化范围

Day 12 对智能体做展示前优化：

1. 优化 `LLM_ENABLED=false` 时的规则版回答。
2. fallback 回答更明确说明大模型暂不可用，并继续给出排查建议。
3. 根据离线状态、未处理告警、光照阈值、最新光照和最近控制日志生成更具体的运维建议。
4. 补充智能体测试问题、项目演示脚本和答辩说明。

## 不允许智能体直接执行的操作

智能体不能直接：

1. 开灯、关灯或设置亮度。
2. 修改阈值配置。
3. 处理告警。
4. 删除设备或业务数据。
5. 修改数据库。
6. 替代用户执行控制命令。

当前智能体只给排查建议。

## 接口说明

### POST /api/agent/chat

请求字段：

| 字段 | 类型 | 必填 | 说明 |
| --- | --- | --- | --- |
| question | string | 是 | 用户问题，不能为空，最多 500 字 |
| device_id | integer | 否 | 设备 ID |
| device_code | string | 否 | 设备编码，例如 `SL-001` |

规则：

- 同时传 `device_id` 和 `device_code` 时，后端会校验两者是否对应同一设备。
- 设备不存在时返回 `404`。
- `question` 为空或超过 500 字时返回 `400`。
- LLM 调用失败时不返回 `500`，而是返回 fallback 回答。

## 请求示例

按 `device_id` 提问：

```json
{
  "question": "这个设备最近为什么离线？",
  "device_id": 1
}
```

按 `device_code` 提问：

```json
{
  "question": "SL-001 最近光照是否异常？",
  "device_code": "SL-001"
}
```

不指定设备，询问系统整体情况：

```json
{
  "question": "当前系统有哪些需要关注的问题？"
}
```

## 响应示例

```json
{
  "answer": "SL-001 当前状态为 offline，最近一次心跳时间较早，系统已生成离线告警。建议先检查设备供电、网络连接和 MQTT 心跳上报程序。",
  "source": "mock",
  "related_device": {
    "id": 1,
    "device_code": "SL-001",
    "device_name": "一号路灯"
  },
  "context_summary": {
    "has_device": true,
    "latest_light_found": true,
    "alarm_count": 1,
    "control_log_count": 3,
    "unhandled_alarm_count": 1
  }
}
```

`source` 说明：

| 值 | 说明 |
| --- | --- |
| `mock` | `LLM_ENABLED=false`，使用规则版回答 |
| `llm` | 成功调用大模型 |
| `fallback` | 启用了 LLM，但配置缺失或调用失败，返回规则版兜底回答 |

## 上下文数据来源

设备维度上下文：

1. `devices`：设备基础信息、状态、最后心跳时间。
2. `light_data`：最新光照、最近 5 条光照历史。
3. `threshold_configs`：阈值配置。
4. `control_logs`：最近 5 条控制日志。
5. `alarm_logs`：最近 5 条告警日志、未处理告警数量。

系统维度上下文：

1. 设备总数。
2. 在线设备数。
3. 离线设备数。
4. 未处理告警数量。
5. 最近 5 条告警。
6. 最近 5 条控制日志。

## LLM 配置说明

配置位于 `backend/.env`：

```env
LLM_ENABLED=false
LLM_PROVIDER=openai-compatible
LLM_API_KEY=
LLM_BASE_URL=
LLM_MODEL=
LLM_TIMEOUT_SECONDS=30
```

注意：

1. `.env.example` 和 README 中不能写真实 API Key。
2. 真实 API Key 只放在本地 `.env`。
3. 后端不会在响应中返回 API Key。
4. 后端不会在日志中打印 API Key。
5. 未配置 LLM 时，后端仍可正常启动。

## 本地演示方式

本地不配置大模型时：

```env
LLM_ENABLED=false
```

启动后端后调用：

```text
POST /api/agent/chat
```

接口会根据数据库上下文返回规则版回答，`source=mock`。

## 大模型调用方式

当：

```env
LLM_ENABLED=true
```

后端会使用 OpenAI-compatible chat completions 风格调用：

```text
{LLM_BASE_URL}/chat/completions
```

因此配置 OpenAI-compatible 服务时，`LLM_BASE_URL` 应填写到 `/v1`：

```env
LLM_BASE_URL=https://api.openai.com/v1
```

不要写成：

```env
LLM_BASE_URL=https://api.openai.com/v1/chat/completions
```

因为代码会自动追加 `/chat/completions`。

请求中包含：

- `model`
- `messages`
- `temperature`

如果 Base URL、API Key、Model 缺失或调用失败，接口返回 `source=fallback`。

## Mock/Fallback 回答规则

本地规则版回答会根据上下文给出建议：

1. 设备 `offline`：建议检查供电、网络、设备网关和 MQTT status 心跳程序。
2. 有未处理离线告警：提醒优先查看和处理离线告警。
3. 最新光照低于低阈值：说明系统建议开灯。
4. 最新光照高于高阈值：说明系统建议关灯。
5. 没有最新光照：提醒检查 telemetry 上报。
6. 有最近控制日志：说明最近一次命令和记录结果。
7. 系统整体问题：汇总设备总数、在线数、离线数和未处理告警数。

回答只会说“建议检查”“建议执行”“可以通过手动控制接口”，不会声称已经执行设备控制。

## System Prompt 约束

给大模型的系统提示要求：

1. 你是智慧路灯节能系统的运维问答助手。
2. 只能根据后端提供的上下文回答。
3. 如果上下文不足，要明确说明“当前数据不足，无法确定”。
4. 不要编造不存在的设备、告警、光照数据。
5. 不要声称已经执行了开灯、关灯、修改阈值等操作。
6. 只能给排查建议，不能代替用户执行控制命令。
7. 回答要简洁，适合项目演示和运维人员阅读。
8. 优先用中文回答。
