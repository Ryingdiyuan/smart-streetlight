# 答辩说明

## 1. 项目背景

智慧路灯节能系统面向校园和园区路灯节能管理场景。传统路灯管理依赖人工巡检，难以及时发现设备离线、光照异常和控制记录问题。本项目通过后端服务、MQTT 消息链路和数据库持久化，模拟智慧路灯的设备接入、数据采集、控制和告警流程。

## 2. 系统目标

1. 管理路灯设备基础信息。
2. 采集设备光照、电压和灯状态数据。
3. 根据光照阈值给出开灯或关灯建议。
4. 支持人工下发控制命令。
5. 通过心跳检测设备在线状态。
6. 对离线设备生成告警。
7. 通过智能体问答辅助运维排查。

## 3. 后端架构

后端采用 FastAPI 分层结构：

```text
routers   接口层，处理 HTTP 请求
schemas   请求和响应数据结构
models    SQLAlchemy 数据库模型
services  业务逻辑和上下文聚合
mqtt      MQTT 客户端和消息处理
tasks     定时任务
core      配置和数据库连接
```

这种结构便于后续扩展，例如新增更多告警类型、设备控制回执或智能体能力。

## 4. 数据流说明

### HTTP 数据流

前端或 Swagger 调用 HTTP 接口，后端通过 SQLAlchemy 操作 MySQL，例如设备 CRUD、光照上报、阈值配置和告警查询。

### MQTT telemetry 数据流

设备或 MQTTX 发布：

```text
streetlight/{deviceCode}/telemetry
```

后端订阅后解析 payload，写入 `light_data` 表，并触发阈值判断。

### MQTT status 数据流

设备发布：

```text
streetlight/{deviceCode}/status
```

后端更新设备 `status` 和 `last_heartbeat_at`。定时任务根据心跳时间判断是否离线。

### MQTT command 数据流

用户调用手动控制接口后，后端记录 `control_logs`，并向：

```text
streetlight/{deviceCode}/command
```

发布控制命令。

## 5. MQTT 为什么需要 Broker

MQTT Broker 是消息中转中心。设备端和后端不需要直接建立点对点连接，而是通过 topic 发布和订阅消息。

优点：

1. 解耦设备和后端。
2. 支持多个设备同时接入。
3. 支持后端订阅通配 topic，例如 `streetlight/+/telemetry`。
4. 适合物联网场景中低带宽、轻量级消息传输。

本项目本地使用 Mosquitto 作为 MQTT Broker。

## 6. 数据库表说明

| 表名 | 说明 |
| --- | --- |
| `devices` | 设备基础信息、状态、最后心跳时间 |
| `light_data` | 光照、电压、灯状态等遥测数据 |
| `threshold_configs` | 设备阈值配置 |
| `control_logs` | 手动控制命令日志 |
| `alarm_logs` | 离线告警和处理状态 |

当前使用 `create_all` 创建不存在的表，但它不会自动修改已有表字段。已有旧表需要手动执行 `ALTER TABLE`。

## 7. 核心接口说明

| 模块 | 接口示例 | 说明 |
| --- | --- | --- |
| 健康检查 | `GET /api/health` | 判断后端是否运行 |
| 设备管理 | `POST /api/devices` | 创建设备 |
| 光照数据 | `POST /api/devices/{id}/light-data` | HTTP 模拟光照上报 |
| 阈值配置 | `PUT /api/devices/{id}/threshold` | 修改判断阈值 |
| 控制命令 | `POST /api/devices/{id}/commands` | 手动控制命令 |
| 告警管理 | `GET /api/alarms` | 查询告警 |
| 智能体问答 | `POST /api/agent/chat` | 运维问答建议 |

## 8. 智能体问答如何减少幻觉

### 只基于数据库上下文回答

智能体的上下文来自后端查询结果，包括设备、光照、阈值、控制日志和告警日志。提示词要求它只能根据这些数据回答。

### 不直接执行设备控制

智能体不会调用开灯、关灯、修改阈值、处理告警等接口，只能给建议。

### 上下文不足时说明数据不足

如果没有最新光照或没有设备数据，回答会说明当前数据不足，建议先检查数据上报。

### LLM 失败时 fallback

当 `LLM_ENABLED=true` 但配置缺失、接口超时或调用失败时，后端不会崩溃，而是返回规则版 fallback 回答。

## 9. 当前不足

1. 没有前端正式联调。
2. 没有 `command/reply` 回执处理。
3. 没有 Alembic 数据库迁移。
4. 没有登录鉴权。
5. 智能体还不是多轮 RAG。
6. 告警类型目前较少，主要是离线告警。

## 10. 后续优化方向

1. 接入前端页面，完成完整联调。
2. 接入真实硬件设备和传感器。
3. 增加 `command/reply`，确认设备是否真正执行命令。
4. 引入 Alembic 管理数据库迁移。
5. 增加登录鉴权和角色权限。
6. 扩展智能体多轮对话、知识库 RAG 和更细的告警诊断。
7. 增加电压异常、光照异常、控制失败等告警类型。
