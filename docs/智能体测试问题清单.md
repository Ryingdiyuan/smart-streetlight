# 智能体测试问题清单

## 测试前准备

建议先准备一台设备，例如 `SL-001`：

1. `POST /api/devices` 创建设备。
2. `POST /api/devices/{device_id}/light-data` 上报至少一条光照数据。
3. `PUT /api/devices/{device_id}/threshold` 配置阈值。
4. `POST /api/devices/{device_id}/commands` 生成一条控制日志。
5. 可通过 MQTT status 或等待离线检测生成告警。

本地演示建议：

```env
LLM_ENABLED=false
```

这样不需要真实大模型 API，也能返回规则版运维建议。

## 设备维度问题

### 1. SL-001 最近为什么离线？

请求示例：

```json
{
  "question": "SL-001 最近为什么离线？",
  "device_code": "SL-001"
}
```

预期回答方向：

- 说明设备当前状态。
- 如果状态为 `offline`，建议检查供电、网络、设备网关、MQTT status 心跳程序。
- 如果存在未处理离线告警，提醒先处理离线告警。

是否需要准备数据：需要已有 `SL-001` 设备；最好有离线状态或离线告警。

### 2. 这个设备最近有没有告警？

请求示例：

```json
{
  "question": "这个设备最近有没有告警？",
  "device_id": 1
}
```

预期回答方向：

- 返回未处理告警数量。
- 简要说明最近告警类型。
- 建议进入告警列表确认详情。

是否需要准备数据：需要设备和告警数据。

### 3. 这个设备最新光照是多少？

请求示例：

```json
{
  "question": "这个设备最新光照是多少？",
  "device_code": "SL-001"
}
```

预期回答方向：

- 说明最新光照强度。
- 结合阈值说明是否建议开灯、关灯或保持。
- 如果没有最新光照，提示检查 telemetry 上报。

是否需要准备数据：需要至少一条 `light_data`。

### 4. 这个设备最近有没有人工控制？

请求示例：

```json
{
  "question": "这个设备最近有没有人工控制？",
  "device_code": "SL-001"
}
```

预期回答方向：

- 说明最近一次控制命令和结果。
- 如果没有控制日志，说明最近没有查到人工控制记录。

是否需要准备数据：建议先调用一次手动控制接口。

### 5. 这个设备阈值设置是否合理？

请求示例：

```json
{
  "question": "这个设备阈值设置是否合理？",
  "device_code": "SL-001"
}
```

预期回答方向：

- 说明低阈值和高阈值。
- 结合最新光照说明当前建议。
- 如果阈值判断关闭，说明系统暂不生成建议。

是否需要准备数据：需要阈值配置和光照数据。

## 系统维度问题

### 1. 当前系统有哪些设备需要关注？

请求示例：

```json
{
  "question": "当前系统有哪些设备需要关注？"
}
```

预期回答方向：

- 汇总设备总数、在线数、离线数、未处理告警数。
- 如果存在离线设备或未处理告警，建议优先关注。

是否需要准备数据：建议已有多个设备和至少一条告警。

### 2. 当前有多少未处理告警？

请求示例：

```json
{
  "question": "当前有多少未处理告警？"
}
```

预期回答方向：

- 返回未处理告警数量。
- 建议进入告警列表查看详情。

是否需要准备数据：可选。

### 3. 哪些设备处于离线状态？

请求示例：

```json
{
  "question": "哪些设备处于离线状态？"
}
```

预期回答方向：

- 当前 MVP 的系统上下文包含离线数量，不列出所有设备明细。
- 如果需要设备明细，建议查询设备列表。

是否需要准备数据：需要设备数据。

### 4. 最近有没有控制记录？

请求示例：

```json
{
  "question": "最近有没有控制记录？"
}
```

预期回答方向：

- 说明最近控制命令和结果。
- 如果没有记录，说明没有查到最近控制记录。

是否需要准备数据：建议先调用手动控制接口。

### 5. 系统当前运行是否正常？

请求示例：

```json
{
  "question": "系统当前运行是否正常？"
}
```

预期回答方向：

- 汇总设备在线/离线状态。
- 汇总未处理告警数量。
- 给出继续观察 telemetry、status、告警列表的建议。

是否需要准备数据：建议已有设备、光照、告警数据。

## 异常测试

### 1. 空 question

请求示例：

```json
{
  "question": ""
}
```

预期结果：返回 `400`，提示 `question 不能为空`。

### 2. 超长 question

请求示例：`question` 超过 500 字。

预期结果：返回 `400`，提示 `question 不能超过 500 字`。

### 3. 不存在的 device_id

请求示例：

```json
{
  "question": "这个设备怎么了？",
  "device_id": 999999
}
```

预期结果：返回 `404`，提示设备不存在。

### 4. 不存在的 device_code

请求示例：

```json
{
  "question": "这个设备怎么了？",
  "device_code": "NOT-EXIST"
}
```

预期结果：返回 `404`，提示设备不存在。

### 5. LLM_ENABLED=false

配置：

```env
LLM_ENABLED=false
```

预期结果：接口返回 `source=mock`，回答由本地规则生成。

### 6. LLM_ENABLED=true 但 API Key 缺失

配置：

```env
LLM_ENABLED=true
LLM_API_KEY=
```

预期结果：接口返回 `source=fallback`，不崩溃。

### 7. LLM 调用失败 fallback

配置错误的 `LLM_BASE_URL` 或关闭网络。

预期结果：接口返回 `source=fallback`，回答中说明大模型暂不可用，并给出规则版建议。
