# 模拟器使用说明

## 1. 文档用途

本文档用于指导项目组成员在自己的电脑上启动并使用“传感器模拟控制台”，完成以下工作：

1. 配置 MQTT Broker 连接信息。
2. 在前端页面中管理模拟设备。
3. 模拟设备发送 `status` 和 `telemetry` 数据。
4. 让前端总览、设备详情、实时光照、历史光照等页面展示真实联调数据。
5. 观察控制命令、日志、告警等闭环效果。

当前模拟器已经内置在项目后端中，不需要额外单独运行 `mock_mqtt_device.py` 才能使用页面版模拟器。

---

## 2. 使用前提

使用模拟器前，本机需要具备以下环境：

- Windows PowerShell
- Python 3.11 及以上
- Node.js 18 及以上
- MySQL
- 一个可用的 MQTT Broker
  - 可以使用本地 Mosquitto
  - 也可以使用 EMQX Cloud

项目目录默认为：

```text
F:\smart-streetlight\smart-streetlight
```

## 4. 首次使用配置

### 4.1 配置后端 `.env`

先检查 `backend/.env` 是否存在，并确认以下配置：

```env
MQTT_ENABLED=true
MQTT_HOST=你的Broker地址
MQTT_PORT=1883
MQTT_USERNAME=你的账号
MQTT_PASSWORD=你的密码
MQTT_CLIENT_ID=smart-streetlight-backend

SCHEDULER_ENABLED=true
DEVICE_OFFLINE_SECONDS=30
```

说明：

- `MQTT_ENABLED=true`：开启 MQTT 联调。
- `MQTT_HOST` / `MQTT_PORT`：Broker 地址和端口。
- `MQTT_USERNAME` / `MQTT_PASSWORD`：如果 Broker 需要鉴权，则必须填写。
- `DEVICE_OFFLINE_SECONDS=30`：演示时推荐设置较短，方便快速看到离线告警。

### 4.2 配置前端 `.env.local`

检查 `frontend/.env.local`：

```env
VITE_SERVICE_MODE=api
VITE_API_BASE_URL=/api
VITE_API_PROXY_TARGET=http://127.0.0.1:8000
```

说明：

- `VITE_SERVICE_MODE=api`：前端走真实后端接口。
- `VITE_API_PROXY_TARGET`：本地开发时代理到后端。

## 5. 启动步骤

模拟器页面依赖“后端 + 前端”两个服务同时运行。

### 5.1 启动后端

在 PowerShell 中执行：

```powershell
cd F:\smart-streetlight\smart-streetlight\backend
python -m uvicorn app.main:app --reload --host 127.0.0.1 --port 8000
```

如果启动成功，应看到：

```text
Uvicorn running on http://127.0.0.1:8000
```

### 5.2 启动前端

在另一个 PowerShell 中执行：

```powershell
cd F:\smart-streetlight\smart-streetlight\frontend
npm.cmd run dev
```

如果启动成功，应看到类似：

```text
Local: http://127.0.0.1:5173/
```

### 5.3 初始化管理员账号

如果本机数据库还没有用户，先执行：

```powershell
$body = @{ username = "admin"; password = "123456" } | ConvertTo-Json
Invoke-RestMethod "http://127.0.0.1:8000/api/auth/init-admin" -Method Post -ContentType "application/json" -Body $body
```

## 6. 页面使用步骤

### 6.1 登录系统

浏览器打开前端地址，例如：

```text
http://127.0.0.1:5173
```

使用以下账号登录：

- 用户名：`admin`
- 密码：`123456`

### 6.2 打开模拟器控制台

登录后点击左侧菜单：

```text
模拟器控制台
```

或直接访问：

```text
http://127.0.0.1:5173/simulator
```

### 6.3 配置 Broker

在页面上方的 `Broker 配置` 中填写：

- MQTT Host / IP
- MQTT 端口
- 账号
- 密码
- 是否启用 MQTT 模拟发送

填写完成后点击：

```text
保存配置
```

保存成功后，页面会显示：

- 当前连接状态
- Client ID

如果显示“未连接”，优先检查：

1. Broker 地址是否正确。
2. 端口是否正确。
3. 账号密码是否正确。
4. 当前电脑是否能访问该 Broker。

### 6.4 查看数据库设备

模拟器控制台的设备列表来自数据库 `devices` 表。

也就是说：

- 数据库里已有设备，会自动显示在页面里。
- 进入设备详情页后，左侧菜单仍归属“设备列表”模块。

### 6.5 新增传感器

在 `新增传感器` 区块中填写：

- 设备编码
- 设备名称
- 安装位置
- 基础光照
- 波动范围
- 基础电压
- 遥测间隔
- 心跳轮次
- 默认在线
- 是否创建后立即开始模拟

点击：

```text
新增传感器
```

效果：

1. 新设备会写入数据库。
2. 新设备会进入模拟器控制表。
3. 如果勾选“立即开始模拟”，会自动开始发 MQTT 数据。

### 6.6 控制单个设备

在设备列表中，每台设备都支持：

- `启动`
- `停止`
- `编辑`
- `删除`

#### 启动

点击 `启动` 后，该设备会开始周期发送：

- `streetlight/{deviceCode}/status`
- `streetlight/{deviceCode}/telemetry`

#### 停止

点击 `停止` 后，该设备停止发送模拟数据。

#### 编辑

点击 `编辑` 后，可调整：

- 设备名称
- 安装位置
- 基础光照
- 波动范围
- 基础电压
- 遥测间隔
- 心跳轮次
- 在线状态
- 保存后是否继续运行

#### 删除

点击 `删除` 后，会：

1. 从模拟器控制台移除该设备。
2. 同步删除数据库中的对应设备记录。

## 7. 数据如何流转

页面版模拟器的完整链路如下：

1. 在模拟器控制台中启动某个设备。
2. 后端模拟器服务按间隔发送 MQTT 消息。
3. 后端业务 MQTT 客户端订阅并接收这些消息。
4. `status` 更新 `devices.status` 和 `devices.last_heartbeat_at`。
5. `telemetry` 写入 `light_data`。
6. 前端业务页面再通过真实接口读取这些数据。

对应主题如下：

| Topic | 方向 | 说明 |
| --- | --- | --- |
| `streetlight/{deviceCode}/status` | 模拟器 -> 后端 | 上报心跳和在线状态 |
| `streetlight/{deviceCode}/telemetry` | 模拟器 -> 后端 | 上报光照、电压、灯状态 |
| `streetlight/{deviceCode}/command` | 后端 -> 模拟器 | 前端手动开灯/关灯后下发控制命令 |

## 8. 如何验证模拟器是否生效

### 8.1 页面内验证

可直接观察以下页面：

- `/simulator`
- `/`
- `/devices`
- `/devices/{id}`
- `/realtime-light`
- `/light-history`
- `/alarms`

建议重点看：

1. 模拟器控制台中的运行状态和日志。
2. 总览大屏中的设备数量、在线设备、最近控制记录。
3. 实时光照页中的数值变化。
4. 历史光照页中的曲线点位增长。
5. 设备详情页中的最近心跳、当前光照、控制日志。

### 8.2 接口验证

可以通过以下接口确认数据是否进入后端：

```powershell
$body = @{ username = "admin"; password = "123456" } | ConvertTo-Json
$login = Invoke-RestMethod "http://127.0.0.1:8000/api/auth/login" -Method Post -ContentType "application/json" -Body $body
$headers = @{ Authorization = "Bearer $($login.access_token)" }
```

查询设备列表：

```powershell
Invoke-RestMethod "http://127.0.0.1:8000/api/devices" -Headers $headers
```

查询某台设备最新光照：

```powershell
Invoke-RestMethod "http://127.0.0.1:8000/api/devices/1/latest-light" -Headers $headers
```

查询某台设备光照历史：

```powershell
Invoke-RestMethod "http://127.0.0.1:8000/api/devices/1/light-history" -Headers $headers
```

查询告警：

```powershell
Invoke-RestMethod "http://127.0.0.1:8000/api/alarms" -Headers $headers
```

## 9. 运行日志说明

模拟器控制台下方的 `运行日志` 用于展示：

- 模拟器连接 Broker
- 发送 `status`
- 发送 `telemetry`
- 收到 `command`
- 启动设备
- 停止设备
- 移除设备
- MQTT 连接失败等异常

当前日志支持：

- 按 `INFO / WARN / ERROR` 级别筛选
- 清空日志
- 面板内滚动查看更多历史记录

说明：

- 页面默认展示最近一段日志内容。
- 如果需要查看更多旧日志，请使用日志区滚动条。

## 10. 常见问题排查

### 10.1 页面显示未连接

优先检查：

1. `MQTT_ENABLED` 是否为 `true`
2. Broker 地址是否可达
3. 端口是否正确
4. 账号密码是否正确

### 10.2 页面设备已启动，但没有业务数据

优先检查：

1. 后端是否真的在运行
2. Broker 是否连接成功
3. 设备编码是否与数据库 `device_code` 一致
4. 前端是否运行在 `api` 模式

### 10.3 前端打开报 404 或无法登录

优先检查：

1. 后端是否运行在 `127.0.0.1:8000`
2. 前端是否运行在 `5173`
3. `frontend/.env.local` 是否指向了正确后端地址
4. 数据库中是否已初始化管理员账号

### 10.4 点击开灯/关灯没有效果

优先检查：

1. 模拟器对应设备是否处于“运行中”
2. Broker 是否正常连接
3. 模拟器日志中是否出现“收到命令”
4. 设备详情页控制日志里的结果是 `success`、`failed` 还是 `skipped`

含义如下：

- `success`：命令已成功发布并被模拟器接收
- `failed`：发布失败，通常是 MQTT 连接异常
- `skipped`：当前进程未启用 MQTT

## 11. 推荐使用流程

组员第一次在自己电脑上使用时，建议按以下顺序操作：

1. 配置 `backend/.env`
2. 配置 `frontend/.env.local`
3. 启动后端
4. 启动前端
5. 初始化管理员
6. 登录系统
7. 打开 `/simulator`
8. 保存 Broker 配置
9. 启动 1 台设备进行验证
10. 再逐步新增其他设备
11. 打开总览、设备详情、实时光照、历史光照页面进行联调确认

---

## 12. 结论

页面版模拟器已经可以作为组内统一的联调工具使用。  
组员无需手工构造 MQTTX 消息，也无需单独跑外部模拟脚本，只需要：

1. 保证本地数据库、后端、前端可运行
2. 保证 MQTT Broker 可连接
3. 在 `/simulator` 页面中完成设备管理和模拟控制

这样就可以在自己的电脑上复现完整的“设备模拟 -> MQTT -> 后端入库 -> 前端展示”闭环。
